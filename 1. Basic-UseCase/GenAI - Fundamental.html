1. Tokens — The Building Blocks

Tokens are the fundamental units of text processing in Large Language Models. They are not exactly words but rather chunks of text that models use as their basic input and output units. Understanding tokens is critical because pricing, rate limits, and context windows all depend on token counts.

A useful rule of thumb: one token is roughly ¾ of an English word. This means 100 tokens translate to about 75 words. However, this varies by language and tokenization method. Different models use different tokenizers:

  • OpenAI (GPT): Uses cl100k_base with ~100,000 token vocabulary
  • Claude (Anthropic): Uses custom BPE-based tokenizer
  • Open-source models: Often use GPT-2 tokenizer or SentencePiece

import tiktoken

enc = tiktoken.get_encoding("cl100k_base")
tokens = enc.encode("Hello, world!")
print(tokens)  // [9906, 11, 1917, 0]
print(len(tokens))  // 4 tokens

// "RAG systems combine retrieval and generation"

💡 Best Practice: Always reserve 10-20% of your context window for the model's response. If you have a 128K context limit, aim to keep input under 100K tokens to leave room for generation. Use token counting libraries like tiktoken (OpenAI) or transformers (Hugging Face) before sending requests.

2. Temperature — Controlling Creativity

Temperature is a parameter that controls the randomness and creativity of model outputs. It works by scaling the logits before applying the softmax function — essentially reshaping the probability distribution of possible next tokens.

Temperature RangeBehaviorBest For
0.0 – 0.3Deterministic, focused, consistentCode generation, data extraction, factual QA
0.4 – 0.7Balanced, naturally creativeGeneral conversation, content writing, summarization
0.8 – 1.2Highly creative, varied, surprisingStorytelling, brainstorming, poetry, ideation
 
// Temperature effect visualization
T=0.0: "The cat sat on the mat."        // deterministic
T=0.5: "The fluffy cat sat on the warm mat."  // balanced
T=1.0: "A sleepy feline perched atop the woven rug." // creative
📊 Top P (Nucleus Sampling): An alternative to temperature. It selects from the smallest set of tokens whose cumulative probability exceeds a threshold (e.g., top_p=0.9). Often used in combination with temperature for more controlled randomness. For most production applications, temperature between 0.1 and 0.5 offers the best balance of reliability and natural language flow.

3. Hallucination — The Accuracy Challenge

Hallucination occurs when LLMs generate plausible-sounding but factually incorrect or nonsensical information. This is one of the fundamental challenges in deploying LLMs in production. Even state-of-the-art models like GPT-4 hallucinate approximately 15-20% of the time on complex factual queries.

Types of Hallucinations

  • Factual hallucinations: Incorrect dates, names, events, or statistics
  • Logical hallucinations: Contradictory reasoning or impossible scenarios
  • Contextual hallucinations: Ignoring or misinterpreting provided context
  • Extrinsic hallucinations: Adding details not present in source material
⚠️ Critical Warning: Never trust LLM outputs for critical decisions without validation. For production applications, always combine LLM generation with verification layers, citation systems, and human-in-the-loop for high-stakes decisions.

Mitigation Strategies

  • RAG (Retrieval-Augmented Generation): Ground responses in retrieved documents
  • Lower temperature: Reduce randomness for factual tasks
  • System prompts: "Only answer if you are confident" or "If uncertain, say 'I don't know'"
  • Chain of Thought: Force reasoning steps before final answer
  • Self-consistency: Generate multiple responses and vote
  • Output validation: Use secondary models or rule-based checks

4. Context Window — Memory Capacity

The context window defines the total number of tokens a model can process in a single interaction — including both input and output tokens. This is essentially the model's "working memory." Different models offer vastly different capacities:

ModelContext Window≈ Pages of Text
GPT-3.5 Turbo16K tokens~24 pages
GPT-4 Turbo128K tokens~192 pages
Claude 3 Opus/Sonnet200K tokens~300 pages
Gemini 1.5 Pro1M tokens~1,500 pages
Llama 38K – 128K tokensVaries by variant

Context Management Strategies

  • Sliding Window: Keep only recent conversation history, drop older messages
  • Summarization: Compress old context into concise summaries
  • RAG: Retrieve only relevant context instead of including everything
  • Chunking: Break long documents into manageable pieces (512-1024 tokens with 10-20% overlap)
  • Hierarchical Summarization: Summarize sections, then summarize summaries
💡 Golden Rule: Always leave room for output tokens. A good rule of thumb is to reserve 10-20% of your context window for the model's response to prevent truncation.

5. Prompt Design — The CO-STAR Framework

Prompt engineering is the art and science of crafting effective inputs to get desired outputs from LLMs. The CO-STAR framework provides a structured approach to prompt design:

  • C (Context): Background information and relevant details
  • O (Objective): Clear goal of what the prompt should achieve
  • S (Style): Desired tone, format, and presentation
  • T (Tone): Attitude and voice (professional, friendly, authoritative)
  • A (Audience): Who the output is intended for
  • R (Response): Desired output structure and format
<system>
You are a technical documentation writer specializing in AI systems.
</system>

<task>
Explain RAG (Retrieval-Augmented Generation) to beginners.
</task>

<constraints>
- Use simple analogies
- Keep under 100 words
- Include a real-world example
</constraints>

<format>
Return as markdown with bullet points.
</format>
🎯 Key Components of Effective Prompts: Role/Persona assignment, clear task description, relevant context, explicit constraints (length, format, style), and optional examples.

6. Few-Shot Learning — Examples as Instructions

Few-shot learning involves providing examples within the prompt to demonstrate the desired pattern or behavior. This technique dramatically improves accuracy by showing the model exactly what you want.

Types of Shot Learning

  • Zero-shot: No examples, just instruction
  • One-shot: Single example
  • Few-shot: 2-10 examples (optimal for most use cases)
  • Many-shot: 10+ examples (enabled by large context windows)
// Few-shot example: Sentiment classification

Input: "I absolutely love this product!" → Positive
Input: "This is the worst experience ever." → Negative
Input: "It's okay, nothing special." → Neutral
Input: "Could not be happier with my purchase!" → Positive
Input: "This is terrible quality." → 

// Model predicts: Negative
📊 Research Insight: Few-shot prompting improves accuracy by 20-40% on complex tasks compared to zero-shot. Best practices include using diverse examples that cover edge cases, formatting examples consistently, and including 3-5 examples for optimal results.

7. Formatting & Instructions

Using structure and delimiters organizes prompts for better comprehension. The way you format your prompt significantly impacts output quality.

Formatting Techniques

  • XML tags: <instruction>, <context>, <example> — most reliable across models
  • Delimiters: Triple backticks (```), triple quotes ("""), dashes (---)
  • Markdown: Headers, lists, code blocks for human-readable structure
  • JSON structure: For machine-readable inputs

Instruction Best Practices

  • Role assignment: "Act as a senior software architect"
  • Constraint definition: "Answer in 2-3 sentences only"
  • Process guidance: "Think step-by-step before answering" (Chain of Thought)
  • Negative instructions: "Do NOT include any markdown formatting"
  • Output specification: "Return as valid JSON"
❌ Vague: "Write about AI"

✅ Clear: "Write a 100-word overview of RAG systems for a technical audience.
Include: 1) Definition, 2) Key components, 3) One use case with real-world example.
Use bullet points and maintain a professional tone."
🧠 Chain of Thought (CoT): Instruct the model to reason step-by-step. This technique improves accuracy on complex reasoning tasks by 30-50%. Example: "Let's approach this step by step. First, identify the problem. Second, consider possible solutions. Third, evaluate trade-offs."

8. JSON Outputs — Programmable Data from LLMs

Getting LLMs to return structured JSON data enables reliable programmatic consumption and integration with existing systems.

Methods for Structured Outputs

  • JSON Mode: Parameter that forces valid JSON output (response_format: { "type": "json_object" })
  • Prompt Engineering: Explicit schema definition in the prompt
  • Constrained Decoding: Grammar-based generation (e.g., Guidance, Outlines)

9. Zod Validation — Type-Safe Schema Validation

Zod is a TypeScript-first schema validation library that ensures LLM outputs match expected structure. It provides runtime validation with detailed error messages and automatic TypeScript type inference.

import { z } from 'zod';

// Define schema
const UserResponseSchema = z.object({
  name: z.string().min(1).max(100),
  email: z.string().email(),
  age: z.number().int().positive().max(120).optional(),
  role: z.enum(['admin', 'user', 'guest']),
  skills: z.array(z.string()).max(10),
  metadata: z.object({
    confidence: z.number().min(0).max(1),
    sources: z.array(z.string()).optional()
  }).optional()
});

// Validate LLM output
const result = UserResponseSchema.safeParse(llmOutput);

if (!result.success) {
  console.log('Validation errors:', result.error.format());
  // Trigger retry with error feedback
  // OR escalate to human review
} else {
  // Type-safe result.data with full TypeScript inference
  const validatedData = result.data;
}
✨ Zod Features: Type inference (TypeScript types automatically derived from schemas), runtime validation with detailed errors, complex validations (email, URL, regex), transform and refine capabilities, and support for nested objects, arrays, and unions.

10. Validation & Retry Strategies

Systematic verification of LLM outputs requires multiple validation layers and robust retry mechanisms for production reliability.

Multi-Layer Validation Pipeline

  • Schema validation: Structure and types (Zod, Pydantic, etc.)
  • Content validation: Factual correctness (knowledge bases, secondary models)
  • Safety validation: Harmful content filtering (profanity, PII, dangerous instructions)
  • Semantic validation: Coherence and relevance to original query
  • Constraint validation: Length, format, style requirements

Retry Strategies

  • Exponential backoff: Increase delay between attempts (1s, 2s, 4s, 8s)
  • Jitter: Add randomness to prevent thundering herd problems
  • Selective retry: Only retry certain errors (429 rate limits, 5xx server errors)
  • Validation retry: Include validation error feedback in next prompt for self-correction
  • Circuit breakers: Stop retrying after repeated failures
async function callWithRetry(fn, maxRetries = 3, baseDelay = 1000) {
  for (let attempt = 0; attempt < maxRetries; attempt++) {
    try {
      const result = await fn();
      if (validate(result)) return result;
      // Validation failed - include errors in next attempt
      throw new Error(`Validation failed: ${JSON.stringify(validationErrors)}`);
    } catch (err) {
      if (attempt === maxRetries - 1) throw err;
      const delay = baseDelay * Math.pow(2, attempt) + Math.random() * 100;
      await new Promise(r => setTimeout(r, delay));
    }
  }
}
⚡ Smart Retry Pattern: When validation fails, include the exact validation errors in the next prompt. This gives the model specific guidance on what to fix, dramatically improving success rates on subsequent attempts — often achieving 95%+ success after 2-3 attempts.

12. Vector Databases — Specialized Storage

Vector databases are specialized systems optimized for storing and searching embedding vectors at scale. They enable fast similarity search across millions or billions of vectors.

Popular Vector Databases

  • Pinecone: Fully managed, high performance, easy to scale
  • Weaviate: Open-source, GraphQL API, built-in modules
  • Chroma: Lightweight, Python-native, great for prototyping
  • Qdrant: Rust-based, high performance, production-ready
  • Milvus: Cloud-native, highly scalable, enterprise features
  • pgvector: PostgreSQL extension, great if already using Postgres

Indexing Algorithms

  • HNSW (Hierarchical Navigable Small World): Fastest search (O(log N)), highest memory usage
  • IVF (Inverted File Index): Balanced performance, memory efficient
  • Flat: Exact search, slowest, best for small datasets (<100K vectors)
// ChromaDB example
import { ChromaClient } from 'chromadb';

const client = new ChromaClient();
const collection = await client.createCollection({
  name: "documents",

13. RAG Architecture — Retrieval-Augmented Generation

RAG (Retrieval-Augmented Generation) combines information retrieval with LLM generation to produce factual, grounded responses. It's the most effective pattern for reducing hallucinations and incorporating domain-specific knowledge.

The Three Stages of RAG

  • Indexing: Documents are chunked → embeddings generated → stored in vector database
  • Retrieval: User query embedded → similarity search → relevant chunks retrieved
  • Generation: Retrieved context augments prompt → LLM generates grounded answer
// RAG Prompt Template
<system>
You are a helpful assistant. Answer based ONLY on the provided context.
If the answer isn't in the context, say "I don't have that information."
Cite sources when possible.
</system>

<context>
{retrieved_chunk_1}
{retrieved_chunk_2}
{retrieved_chunk_3}
</context>

<question>{user_query}</question>
🎯 RAG Benefits: 40-60% reduction in hallucinations compared to standalone LLMs. Enables citation of sources. Updates knowledge without retraining. Handles proprietary/domain-specific data naturally. More cost-effective than fine-tuning.

Advanced RAG Techniques

  • HyDE: Generate hypothetical answer first, then retrieve based on that
  • Self-RAG: Model critiques its own retrieval and generation
  • Multi-step RAG: Iterative retrieval for complex, multi-hop queries
  • Reranking: Use cross-encoders to refine top-k retrieval results
  • Query rewriting: Enhance user queries before embedding for better retrieval

14. Search Strategies & Optimization

Optimizing retrieval is critical for RAG quality. The right search strategy can dramatically improve answer relevance.

Search Types

  • Semantic Search: Vector similarity for meaning-based retrieval (finds conceptually similar content)
  • Keyword Search: BM25, TF-IDF for exact term matching (finds specific phrases)
  • Hybrid Search: Combines both approaches for best results — typically using a weighted combination (alpha parameter balancing semantic vs keyword)
  • Multimodal Search: Extends to images, audio, and other data types

Optimization Techniques

  • Chunking Strategy: 256-1024 tokens with 10-20% overlap maintains coherence across boundaries. For code, consider chunking by functions/classes rather than fixed token counts.
  • Metadata Filtering: Pre-filter by date, source, type before vector search
  • Query Expansion: Rewrite or enhance queries for better retrieval (e.g., generate multiple query variations)
  • Parent Document Retrieval: Retrieve a chunk, return larger parent context for better LLM comprehension
  • Reranking: Apply cross-encoder models to refine top-k results
// Hybrid Search Example
const results = await vectorDB.search({
  query: "How do I implement RAG with streaming?",
  hybrid: { 
    alpha: 0.5,  // 0 = pure keyword, 1 = pure semantic
    weights: { semantic: 0.6, keyword: 0.4 }
  },
  limit: 10,
  filters: { 
    date: { $gte: "2024-01-01" }, 
    source: { $in: ["documentation", "blog"] }
  },
  rerank: true  // Apply cross-encoder reranking
});
💡 Chunking Best Practices: 512 tokens is optimal for most use cases. Add 10-20% overlap between chunks to prevent information loss at boundaries. For code, chunk by functions/classes. For long documents, consider hierarchical chunking (sections first, then paragraphs).
📈 RAG Success Metrics to Track: Precision@k, Recall@k, MRR (Mean Reciprocal Rank), NDCG (Normalized Discounted Cumulative Gain), user satisfaction scores, and hallucination rates. Aim for >85% retrieval relevance and <5% hallucination rate in production.
